# M3 — Figure ↔ 본문 매칭

> **비용 안내: 이 마일스톤은 기본 설정(mock Provider)에서 API 비용이 발생하지 않습니다.**
> 매칭 1단계(명시적 참조 탐지)는 정규식만 쓰는 결정론적 로직이고, 2단계(의미 매칭)의
> 임베딩·LLM 검증은 `app/core/llm/` Provider Layer를 경유합니다 (M1.5). mock/ollama로
> 개발·테스트를 완료할 수 있으며, **Anthropic(유료) 전환 시에만 CLAUDE.md 가드레일에
> 따라 사용자 확인이 필요합니다.**

## 목표

M2가 추출한 각 `Figure`에 대해, **그 Figure를 설명하거나 참조하는 본문 문단 목록**
(`FigureTextMatch`)을 산출한다. 이 결과는 M4(설명 생성)의 근거(citation) 재료가 되므로,
매칭의 **정밀도**(엉뚱한 문단을 근거로 주지 않기)를 재현율보다 우선한다.

## M1/M2에서 넘어온 조건 — 반드시 먼저 읽을 것

두 마일스톤 문서의 "알려진 한계" 절에 기록된 것들이 M3와 직접 맞닿아 있다:

1. **표(Table) 행이 문단으로 저장되어 있음 (M1 한계 #1).** 임베딩 의미 매칭에서 표 조각이
   검색 노이즈가 된다. M3에서는 임베딩 대상 문단을 선별(예: 최소 길이, 표-행 패턴 제외)하는
   필터를 두고, 무엇을 걸렀는지 기록한다. 표 구조의 근본 해결은 여전히 범위 밖.
2. **매칭 인덱스에 없는 Figure가 존재할 수 있음 (M2 한계 #1·#2).** 캡션이 그림 위에 있는
   레이아웃·번호 없는 그림은 추출 단계에서 누락된다. M3는 "존재하는 Figure에 대한 매칭"만
   책임지고, 이 한계를 그대로 승계한다고 명시한다.
3. **`Figure.extraction_confidence`를 매칭 신뢰도에 반영한다 (M2 의존성 절).** conf가 낮은
   Figure(0.3 폴백)는 캡션-이미지 짝 자체가 불확실하므로 매칭 결과에도 그 불확실성이
   전파되어야 한다 (최소한: relevance_score 상한 또는 결과에 플래그).

## 범위

### 포함
- `FigureTextMatch` DB 스키마 + 마이그레이션 (docs/data-model.md 정의 그대로)
- **1단계 — 명시적 참조 탐지 (결정론적)**: 본문에서 "Figure 2", "Fig. 2" 언급을 정규식으로
  찾아 해당 문단을 `match_type="explicit_reference"`로 연결. `quote_span` 기록
- **2단계 — 의미 매칭 (Provider 경유)**: 캡션 임베딩 ↔ 문단 임베딩 유사도로 후보 산출,
  LLM 검증("이 문단이 이 Figure를 설명하는가") 통과 시 `match_type="semantic_similarity"`
- 첫 Agent 폴더 (`grounding_agent`: node.py / prompt.md / schema.py) — docs/adding-an-agent.md
  절차의 첫 실적용
- `POST /papers` 파이프라인에 매칭 단계 통합 (`upload_status="matching"` 경유)
- 매칭 결과 조회 API
- 골든 테스트 (샘플 논문 2편 재사용)

## 작업 지시 (Claude Code에게)

### Step 0. 매칭 골든 케이스 준비 (+ 사람 검수)

M1·M2의 교훈 그대로: 채점 기준을 채점받는 코드가 만들지 않는다.

1. `paper1.pdf`, `paper2.pdf` 본문에서 각 Figure의 **명시적 참조 문단**을 직접 확인한다
   (PDF 원문 검색 + 해당 문단 원문 확인 — 파서 출력에만 의존하지 않는다).
2. `tests/fixtures/sample_papers/paper{N}.matches.expected.json` 형식으로 정리한다:
   ```json
   {
     "note": "판단 근거 기록",
     "figures": [
       {
         "figure_number": "Figure 1",
         "explicit_paragraph_hints": ["shown in Figure 1", "halves of Figure 1"],
         "min_explicit_matches": 1,
         "semantic_hints": ["문단을 특정할 수 있는 고유 문구", "..."]
       }
     ]
   }
   ```
   - `explicit_paragraph_hints`: 그 Figure를 명시 참조하는 문단을 식별하는 부분 문자열
     (문단 id는 업로드마다 바뀌므로 텍스트 앵커로 지정).
   - `semantic_hints`: 명시 참조는 없지만 명백히 그 Figure를 설명하는 문단(있을 때만,
     결정 #1에서 (a)를 선택한 경우의 채점 대상).
3. **초안을 사용자에게 보여주고 검수받은 후에만 Step 1로 진행한다.**

**Step 0 실측 기록 (2026-07-13, 골든 검수 중 사용자 요청으로 재검토):**
- paper1 부록 Figure 3~5는 본문 명시 참조 0건 — 의미 매칭이 유일한 연결 경로.
- 의미 힌트 2건의 실제 임베딩 랭킹(nomic-embed-text, 121개 문단): Fig 5 힌트 **1위**(0.859),
  Fig 3 힌트 **12위**(0.696, 중앙값 0.595).
- **`implicit_reference`(번호 없는 암묵 참조) 카테고리는 도입하지 않기로 판단.** 근거:
  ① 해당 성격의 문장이 전체에서 1건(Fig 5)뿐이고 그마저 임베딩 1위로 잡혀 검색력 기여가
  없음, ② 암묵성 판정은 LLM 의존적이라 결정론적으로 부여할 수 없는 스키마 값이 됨,
  ③ 필요해지면 LLM 검증 출력에 실어 스키마 변경 없이 확장 가능. (data-model.md 무변경)
- 설계 반영: **top-k ≥ 15** (Fig 3이 12위였음), Fig 3 캡션의 상위 5개가 전부 일반
  self-attention 문단이라 **LLM 검증의 정밀도 필터 역할이 실측으로 확인됨**.

### Step 1. 데이터 모델

`docs/data-model.md`의 `FigureTextMatch` 정의 그대로:
- `app/schemas/match.py` (Pydantic) + `app/models/match.py` (ORM — M1 paper.py 패턴,
  FK 인덱스 포함), Alembic 마이그레이션 생성·적용.
- `quote_span: tuple[int,int] | None`은 PG에서 `int[]`로 저장 (bbox와 같은 방식).

### Step 2. 1단계 — 명시적 참조 탐지 (결정론적)

`grounding_agent/node.py` 내부의 **LLM 없는 순수 함수**로 구현 (개발 원칙 5):
- 패턴: `Figure N` / `Fig. N` (캡션 자신은 제외 — 캡션 문단은 매칭 대상이 아님).
- 발견 문단마다 `match_type="explicit_reference"`, `relevance_score=1.0`(정규식 확신),
  `quote_span=(시작, 끝)` 기록.
- M1 한계 대응: 표-행 문단도 "Fig. N"을 포함할 수 있음 — 이 단계에서는 포함하되
  문단 텍스트 길이 등 관찰되는 특성을 Step 0 골든과 대조해 보고.

### Step 3. 첫 Agent 스캐폴드

`docs/adding-an-agent.md` 절차로 `grounding_agent`를 처음 세운다 (복제 원본이 없으므로
이 Agent가 이후 모든 Agent의 원본이 된다):
- `schema.py`: 입력(Figure + 문단 목록) / 출력(FigureTextMatch 목록) Pydantic 모델
- `prompt.md`: 2단계 LLM 검증용 시스템 프롬프트 — 단일 책임("이 문단이 이 Figure를
  설명하는가"만 판단), JSON 출력 형식, "근거 없으면 no로 답할 것" 명시
- `node.py`: 1단계(Step 2) + 2단계(Step 4)를 조립하는 진입 함수. LangGraph 미조립
  상태에서도 API 라우트가 직접 호출 가능한 시그니처로 (M1·M2의 순수 함수 패턴).

### Step 4. 2단계 — 의미 매칭 (Provider 경유)

- 임베딩: `get_embedding_provider()`로 캡션·문단 임베딩 → 코사인 유사도 상위 k 후보
  (명시 참조로 이미 매칭된 문단 제외).
- **임베딩 대상 필터** (M1 한계 대응): 최소 길이 미달·표-행 패턴 문단은 임베딩에서 제외하고,
  걸러진 개수를 실측 보고에 포함.
- LLM 검증: `get_llm_provider()`로 후보별 판정. mock에서는 canned responses로 검증 로직을
  테스트하고, 실품질은 결정 #1의 선택에 따름.
- `relevance_score`: 유사도 기반, `extraction_confidence < 0.5`인 Figure는 상한 적용
  (전파 규칙을 docstring에 기록).
- 임계값(유사도 컷, top-k)은 **추측으로 정하지 말고** 샘플 2편에서 분포를 측정해 근거와
  함께 제안한다 (작업 품질 기준 5).

### Step 5. API + 파이프라인 통합

- `POST /papers` 흐름: figures 저장 후 `upload_status="matching"` → 매칭 실행·저장 → `done`.
  부분 실패 정책(architecture.md 5절): 특정 Figure의 매칭 실패는 그 Figure만 표시하고 계속.
- `GET /papers/{id}/figures/{fid}/matches` — 매칭 문단 목록 (문단 텍스트 포함, score 순).
- 동기 유지 여부는 매칭 포함 전체 처리 시간을 **실측한 뒤** 보고하고 결정 (M2 전례).

### Step 6. 테스트 및 문서 갱신

- 골든: explicit 매칭이 `min_explicit_matches` 이상 + 힌트 문단이 실제로 매칭됐는지.
- 네거티브: 캡션 자신이 매칭에 포함되지 않는지, "Table N" 언급이 Figure 매칭으로 안 새는지.
- API E2E 1개 이상. 전체 회귀(기존 35개) 유지.
- 문서: architecture.md 2.3 확정 내용 반영, data-model.md 정합, README 상태,
  M3.md 하단 "알려진 한계"(+ M4 영향 한 줄).

## M3에서 확인된 알려진 한계 (M4 영향 포함)

M3 완료 시점(2026-07-13) 기준.

1. **의미 매칭의 임베딩 재현율 한계 — 매칭 인덱스에 못 드는 정답 문단이 존재한다.**
   paper1 Figure 3의 정답 문단("path length between long-range dependencies")이 실제
   추출 캡션 기준 임베딩 유사도로 top-20 후보에조차 들지 못했다(순위 20위 밖). 검증기(qwen)
   문제가 아니라 nomic-embed-text의 캡션↔문단 유사도 자체의 한계이며, top-k를 20까지
   넓혀도(C) 복구되지 않았다. → **이것은 M2의 "Figure 자체가 누락되는 경우"(캡션이 그림
   위인 레이아웃·번호 없는 그림)와 같은 계열의 문제다: 파이프라인 어느 단계에서 대상이
   인덱스에 들어오지 못하면, 이후 단계에서 원천적으로 매칭·설명이 불가능한 Figure/문단이
   존재한다.** M2는 "추출 인덱스에 없는 Figure", M3는 "매칭 인덱스에 없는 정답 문단"으로,
   둘 다 "없는 것은 뒤에서 살릴 수 없다"는 동일한 근본 제약이다.
   → **M4 영향: 근거가 명시 참조뿐인 Figure(paper1 Fig 3~5처럼)는 의미 매칭 근거가
   비거나 부족할 수 있다. explainer_agent는 근거 부족을 "본문에 근거 없음"으로 정직하게
   표시해야 하며(architecture.md 3절 grounding 원칙), 근거를 지어내 채우지 않는다.**
   보완 방향(후순위): 임베딩 모델 교체, 캡션 외 Figure 주변 문맥을 질의에 포함, 하이브리드
   검색(BM25 병용) 등. M3에서는 도입하지 않음.

2. **검증기 판단력이 매칭 정밀도의 상한이다.** llama3.2(3B)는 관련성 오판이 잦아
   qwen2.5:7b(7B)로 확정했다(Step 4 실측). 로컬 모델 특성상 재현율은 보수적이다(qwen이
   애매한 후보를 다수 no로 판정). → **M4 영향: relevance_score와 match_type을 M4가 근거
   강도 표시에 쓸 때, semantic_similarity 매칭이 적은 Figure는 근거가 얇을 수 있음을 감안한다.**

## 다음 단계 의존성

M4(설명 생성)는 이 마일스톤의 `FigureTextMatch`(+ 문단 원문)를 근거 재료로 받는다.
`match_type`과 `relevance_score`는 M4가 근거의 강도를 표시하는 데 쓰이므로 성실하게
채운다. 결정 #3에서 그래프를 M4로 미뤘다면, M4 착수 시 state.py 설계에 M3의
node.py 시그니처가 그대로 들어갈 수 있는지 먼저 확인한다.

# M2 — Figure 추출

> **비용 안내: 이 마일스톤은 원칙적으로 Anthropic API 키가 필요 없습니다.**
> PyMuPDF의 임베디드 이미지 추출과 좌표 기반 캡션 매칭 휴리스틱만으로 대부분 해결됩니다.
> 다만 아래 "LLM 보조가 필요할 수 있는 지점"에서 휴리스틱만으로 정확도가 부족하면,
> **코드를 작성하기 전에 먼저 사용자에게 확인**하세요 (`CLAUDE.md` 가드레일 참고). 확인 없이
> API 호출 코드를 추가하지 않습니다.

## 목표

M1이 만든 `Paper`/`Section`/`Paragraph` 구조 위에, 논문에서 **Figure 이미지와 캡션을 추출**하여
저장합니다. 이 결과물(`Figure.image_uri`, `Figure.caption`, `Figure.bbox`)은 M3(Figure↔본문
매칭)의 입력이 되므로, 이번에도 정확성만큼 **참조 가능성**(어느 페이지/좌표에서 왔는지)을
끝까지 유지하는 것이 중요합니다.

이 마일스톤은 Figure의 "의미"를 다루지 않습니다. "이 Figure가 무엇을 그렸는지"는 M4의 일이고,
M2는 순수하게 "Figure 이미지를 정확히 오려내고 캡션과 짝짓는 것"까지만 다룹니다.

## M1에서 넘어온 조건 — 반드시 먼저 읽을 것

`docs/milestones/M1.md`의 "알려진 한계" 절에 아래 2건이 기록되어 있습니다. M2는 이 두 가지와
직접 맞닿아 있으므로, 착수 전에 그 절을 다시 읽고 아래 방침을 따르세요.

1. **표(Table) 행이 문단으로 흡수되는 문제** — M1의 `document_parser`는 표 영역을 텍스트
   블록과 구분하지 않아서, Table 2/3의 행들이 일반 문단처럼 저장되어 있습니다. M2에서 Figure
   영역(이미지)과 표 영역은 다루는 대상이 다르지만, **"이미지가 아닌 그래픽 요소"라는 점에서
   서로 인접한 문제**입니다. M2에서 표를 Figure로 오인해 추출하지 않도록 주의하세요 (Figure는
   임베디드 이미지 객체를 기준으로 판별하므로 보통은 자동으로 구분되지만, 벡터 그래픽으로 그려진
   차트/표가 있다면 애매할 수 있습니다 — 발견되면 M2 결과 보고에 포함하세요).
2. **번호 없는 헤딩(부록 등) 미탐** — Figure도 이와 유사한 문제가 있을 수 있습니다. 캡션이
   "Figure 3"처럼 명확히 번호가 매겨진 경우는 쉽지만, 부록의 그림이나 번호가 없는 다이어그램은
   놓칠 수 있습니다. M1과 마찬가지로 **번호 매김에 의존하는 것을 기본 전략으로 하되, 이 한계를
   숨기지 말고 결과 보고에 명시**하세요.

## 범위

### 포함
- PDF에서 임베디드 이미지(Figure) 추출 + 저장소에 크롭 이미지로 저장
- 캡션 텍스트 추출 및 "Figure N" 패턴 탐지
- 캡션 ↔ 이미지 매칭 (좌표 기반 최근접 휴리스틱)
- `Figure` DB 스키마 + 마이그레이션
- Figure 목록 조회 API
- 골든 테스트 (M1과 같은 샘플 논문 2편 기준 — 재사용)

### 제외 (다음 마일스톤)
- Figure ↔ 본문 문단 매칭 (M3)
- Figure 의미 해석/설명 생성 (M4, Vision 모델 필요)
- 표(Table)를 별도 구조화된 객체로 추출하는 것 — M1 한계로 기록된 표 문제의 "근본 해결"은
  M2 범위가 아닙니다. M2는 표를 Figure로 오인하지 않는 것까지만 책임집니다.
- 수식(Equation) 추출 — 범위 밖 (계획에 없음, 필요시 별도 논의)

### Step 2. `figure_extractor` 서비스 구현

`app/services/figure_extractor.py`에 다음 시그니처로 구현하세요 (M1의 `document_parser`처럼
순수 함수 지향, DB 세션 직접 다루지 않음):

```python
def extract_figures(pdf_path: str) -> list[ExtractedFigure]:
    """PDF에서 Figure 이미지와 캡션을 추출한다.

    반환되는 각 항목은 이미지 바이트, 캡션 텍스트, 페이지, bbox를 포함해야 하며
    (실제 저장은 호출부에서 storage.py를 통해 처리), figure_number는 캡션에서 파싱한
    번호("Figure 1" 등)를 담는다. 파싱 실패 시 "Figure ?N"(발견 순서 기반 임시 번호).
    """
```

> **Step 0 실측 후 전략 수정 (2026-07-12)**: 아래 방침 1은 "임베디드 이미지 순회"를 기본
> 전략으로 제시했으나, 실측 결과 **샘플 논문 12개 Figure 중 임베디드 래스터는 3개뿐**이었다
> (paper1 Fig 1·2만 래스터, paper2는 7개 전부 벡터 그래픽 → `get_images()` 단독 재현율
> paper1 2/5, paper2 0/7). 따라서 **캡션을 앵커로 Figure 영역을 결정하고 페이지를
> 렌더링-크롭하는 방식을 주 전략**으로 하고, 임베디드 이미지는 bbox 결정의 보조 근거 +
> 원본 화질 확보용 빠른 경로로만 사용한다. 상세 알고리즘은 figure_extractor.py docstring 참고.

**구현 방침**:
1. **이미지 추출**: PyMuPDF의 `page.get_images()`로 임베디드 이미지 객체를 페이지별로 순회.
   너무 작은 이미지(예: 아이콘, 로고, 워터마크 추정)는 면적 임계값으로 필터링 — 임계값은
   실제 두 샘플 논문에서 관찰한 값을 근거로 정하고, 그 근거를 docstring에 남기세요.
2. **캡션 탐지**: "Figure N" 또는 "Fig. N" 패턴으로 시작하는 텍스트 블록을 정규식으로 탐지.
   M1의 헤딩 탐지 로직(폰트 스타일 활용)을 참고하되, 캡션은 보통 본문보다 **작은 폰트**이거나
   이탤릭인 경우가 많으므로 그 특징을 활용해도 좋습니다 (관찰 후 필요하면 사용).
3. **이미지-캡션 매칭**: 각 이미지에 대해, 같은 페이지에서 **세로 좌표상 가장 가까운 캡션
   텍스트 블록**을 연결 (보통 이미지 바로 아래). 여러 이미지가 한 페이지에 근접해 있으면
   최근접 매칭이 틀릴 수 있음 — 이 경우를 `extraction_confidence`에 낮은 점수로 반영하세요.
4. **크롭 이미지 저장**: 원본 이미지 바이트를 그대로 저장할지, 아니면 bbox 기준으로 페이지를
   재렌더링해 크롭할지는 실측 후 결정하세요 (임베디드 이미지가 이미 Figure 영역과 정확히
   일치하지 않는 경우가 있을 수 있습니다 — PDF마다 다름). 결정 근거를 docstring에 남기세요.

**LLM 보조가 필요할 수 있는 지점 (착수 전 확인 필요)**:
- 한 페이지에 이미지 여러 개가 근접해 있어 캡션 매칭이 애매한 경우
- 캡션 텍스트가 정규식 패턴에서 벗어난 경우 (드묾)

Step 2 구현 중 이런 케이스가 실제 샘플 논문 2편에서 **한 건이라도 발견되면**, 코드로 LLM 보조를
넣기 전에 먼저 사용자에게 "이런 애매한 케이스가 있는데 API 키가 필요할 수 있다"고 보고하고

## M2에서 확인된 알려진 한계 (M3/M4 영향 포함)

M2 완료 시점(2026-07-12) 기준. 두 샘플 논문에서는 발생하지 않았지만 알고리즘 구조상
존재하는 한계 2건:

1. **캡션이 Figure '위'에 있는 레이아웃 미지원.** bbox 탐색 구역이 캡션 위쪽으로만
   확장된다 (두 샘플 논문 12개 Figure 모두 캡션이 아래에 위치함을 육안 확인하고 결정).
   일부 저널 스타일처럼 캡션이 그림 위에 오는 논문에서는 구역이 비어 conf 0.3 폴백
   (캡션 위 고정 높이 크롭)으로 빠지며, 크롭이 엉뚱한 영역일 수 있다.
   → **M3 영향: 해당 논문에서는 Figure 자체가 누락되거나 잘못 크롭되어 M3 매칭 대상에서
   빠진다.** 필요 시 캡션 아래쪽 구역도 탐색하는 양방향 확장으로 보완 가능 (실패 사례
   확보 후 진행).
2. **번호 없는 그림(캡션 없는 다이어그램, 장식 이미지 등) 미탐.** 캡션 앵커 방식이라

   전략으로 채택). M1의 "번호 없는 헤딩 미탐" 한계와 같은 계열.
   → **M3 영향: 같은 이유로 M3에서 원천적으로 매칭 불가능한 Figure가 존재한다.**

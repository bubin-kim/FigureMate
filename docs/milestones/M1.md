# M1 — PDF 업로드 & 구조 분석

> **비용 안내: 이 마일스톤은 Anthropic API 키가 필요 없습니다.**
> PyMuPDF(로컬 라이브러리)와 정규식/폰트 휴리스틱만으로 구현합니다. `.env`의
> `ANTHROPIC_API_KEY`는 비워둔 채로 M1을 전부 완료할 수 있습니다. LLM 호출 코드를
> 이 마일스톤에 추가하지 마세요 (아래 "구현 방침" 참고).

## 목표

사용자가 PDF를 업로드하면, 시스템이 그 PDF를 **섹션과 문단 단위로 구조화**하여 저장한다.
이 결과물(특히 `Paragraph.id`)은 이후 모든 마일스톤(Figure 추출·매칭·설명)의 기반이 되므로,
정확성보다 **일관성과 참조 가능성**을 우선한다.

이 마일스톤은 Figure를 다루지 않는다. 순수하게 "PDF → 구조화된 텍스트"만 다룬다.

## 범위

### 포함
- FastAPI 프로젝트 초기 스캐폴딩 (`backend/`)
- PDF 업로드 API (`POST /papers`)
- PDF → 섹션/문단 구조화 로직 (`document_parser` 서비스)
- Paper/Section/Paragraph를 저장하는 DB 스키마 + Alembic 마이그레이션
- 처리 상태 조회 API (`GET /papers/{id}`)
- 구조화 결과 조회 API (`GET /papers/{id}/sections`)
- 골든 테스트 (샘플 논문 PDF 2편 기준)

### 제외 (다음 마일스톤)
- Figure 추출/매칭/설명 (M2~M4)
- 프론트엔드 UI (M5)
- 인증/사용자 관리 (범위 밖, 필요 시 별도 계획)
- LangGraph 도입 (M1은 파이프라인의 첫 단계뿐이므로 아직 그래프가 필요 없음. 단, M2~M4에서
  그래프에 편입될 것을 감안해 `document_parser`를 순수 함수로 만들어 재사용 가능하게 한다)

### Step 3. `document_parser` 서비스 구현

`app/services/document_parser.py`에 다음 시그니처의 함수를 구현하세요:

```python
def parse_document(pdf_path: str) -> ParsedDocument:
    """PDF를 섹션과 문단으로 구조화한다.

    반환값의 각 Paragraph는 이후 다른 마일스톤에서 grounding 참조 단위로 쓰이므로,
    text는 원문 그대로(불필요한 정규화 최소화), order는 논문 내 등장 순서를 유지해야 한다.
    """
```

**구현 방침**:
1. PyMuPDF(`fitz`)로 PDF를 열고 페이지별 텍스트 블록을 추출합니다.
2. 섹션 헤딩 탐지: 아래 두 신호를 결합한 휴리스틱을 사용하세요.
   - 정규식 패턴: `^(\d+\.)+\s+[A-Z]` 또는 `^(Abstract|Introduction|Related Work|...)` 같은 관용적 헤딩
   - 폰트 크기: 본문보다 큰 폰트 크기를 가진 텍스트 블록 (PyMuPDF의 `get_text("dict")`로 폰트 정보 획득 가능)
3. 헤딩으로 판별되지 않은 텍스트 블록은 직전 섹션에 속하는 문단으로 취급합니다.
4. 문단 분리는 줄바꿈/들여쓰기 패턴으로 러프하게 나누되, 너무 잘게 쪼개지 않도록 합니다
   (한 문단이 여러 줄인 것은 정상입니다 — 문장 단위가 아니라 논리적 문단 단위입니다).
5. **애매한 경우를 위한 LLM 보조는 이번 마일스톤에서는 구현하지 않습니다.** 휴리스틱만으로
   진행하고, 정확도가 골든 케이스 기준에 못 미치면 그때 보조 로직을 추가하는 방향으로 논의합니다
   (섣불리 LLM을 넣어 비용과 지연을 늘리지 않기 위함).

### Step 4. API 구현

`app/api/routes_papers.py`에 아래 엔드포인트를 구현하세요.

- `POST /papers` — multipart/form-data로 PDF 업로드
  - PDF를 스토리지에 저장(M1 단계에서는 로컬 파일시스템 경로로 충분, 추후 S3로 교체 가능하도록
    `app/services/storage.py`로 인터페이스를 분리해두세요)
  - `Paper` 레코드 생성 (`upload_status="parsing"`)
  - `document_parser.parse_document()` 호출 → `Section`/`Paragraph` 저장
  - 완료 시 `upload_status="done"` (M1 시점에서는 이후 단계가 없으므로 "done"이 최종 상태),
    실패 시 `upload_status="failed"` + `error_message` 기록
  - 응답: 생성된 `Paper`의 id와 status
  - **응답 정책 (구현 시 결정됨)**: 파싱 실패도 업로드가 성공했다면 200 + `upload_status="failed"`
    + `error_message`로 응답한다 (실패는 Paper 리소스의 상태로 표현). 비PDF/빈 파일 등 요청
    자체가 잘못된 경우만 422로 거부한다. `docs/architecture.md` 5절 참고.
  - 처리는 동기로 시작해도 무방합니다 (M1은 순수 텍스트 파싱이라 수 초 내 완료). 비동기 큐는
    M2 이후 Figure/LLM 처리가 붙을 때 다시 검토합니다.

- `GET /papers/{id}` — Paper 상태 조회
- `GET /papers/{id}/sections` — 구조화된 섹션+문단 전체 조회 (중첩 JSON)

### Step 5. 테스트 작성 및 골든 케이스 검증

`tests/fixtures/sample_papers/`에 미리 준비된 PDF와 `*.expected.json`을 사용해:

```python
# tests/services/test_document_parser.py 예시 구조
def test_parse_document_matches_golden(sample_pdf_path, expected_sections):
    result = parse_document(sample_pdf_path)
    # 완전 일치가 아니라 아래 기준으로 채점:
    # - 섹션 개수가 기대값과 ±1 이내
    # - 각 기대 섹션 제목이 결과에서 발견되는가 (fuzzy match 허용)
    # - 전체 문단 수가 너무 적지/많지 않은가 (파싱이 깨지지 않았는지의 신호)
    ...
```

**골든 케이스 채점 기준 (DoD에서 사용)**:
- 섹션 탐지 재현율: 기대 섹션의 90% 이상을 찾아야 함
- 문단 순서: `order` 필드가 원문 순서와 일치해야 함 (섹션 내에서나 전체적으로 역전이 없어야 함)
- API E2E: PDF 업로드 → 상태가 `done`이 될 때까지 → sections 조회까지 한 번에 통과하는
  통합 테스트 최소 1개

## M1에서 발견했고 M2/M3에서 다뤄야 할 알려진 한계

M1 완료 시점(2026-07-12) 골든 케이스 검증 중 발견된 한계 2건 + 이후 마일스톤 작업 중
발견되어 추가된 한계(#3, 2026-07-13):

1. **표(Table) 행이 문단으로 흡수되어 문단 수가 부풀려짐.** 예: paper1(Attention Is All You
   Need)의 Table 2/3 행들이 "5.4 Regularization"(15¶), "6.2 Model Variations"(13¶)의 문단으로
   계상됨. paper2(ResNet)는 표가 많아 4.1절에 140¶이 몰림. → **M2에서 표 영역을 레이아웃
   수준에서 탐지해 문단과 분리해야 함.** M3의 임베딩 매칭 품질에도 영향(표 조각이 검색 노이즈가
   됨)이 있으므로 M2에서 해결하는 것이 순서상 맞다.
2. **번호 없는 부록 헤딩은 섹션으로 탐지되지 않고 직전 섹션에 흡수됨.** 예: paper1의 부록 헤딩
   "Attention Visualizations"(p13)는 번호 매김도 없고 관용적 헤딩 목록에도 없어 References
   섹션의 문단으로 흡수됨. → **번호 매김에 의존하지 않는 헤딩 탐지 보완**(예: 폰트 스타일 +
   레이아웃 위치 기반, 또는 애매한 케이스만 가벼운 LLM 보조 판별 — architecture.md 2.1절의
   유보 조항)이 필요할 수 있음. 관용 헤딩 목록에 단어를 추가하는 방식은 특정 논문 과적합
   위험이 있어 M1에서는 보류함.
3. **2단 컬럼에서 하이픈 분리 단어가 컬럼 경계에 걸리면 병합 문단의 어순이 뒤섞임**
   (M3 Step 2에서 발견, 2026-07-13). 예: paper2 p2의 "feedfor-ward"가 컬럼 경계에서
   분리되면서 병합된 문단 내에서 "…realized by feedfor- Shortcut connections […] are
   those skipping…"처럼 문장 조각이 교차 배치됨 (p3 "Exper-iments"도 동일). 문단이
   담는 내용 자체는 완전하고 Figure 참조 탐지에는 영향이 없으나, **문자열 검색과 인용
   표시 품질이 저하됨**. → **M3 영향: 매칭 골든의 텍스트 앵커는 원문이 아니라 파싱된
   문단 기준으로 선택해야 함 (실제로 힌트 3건 교체됨). M4 영향: 근거 인용문을 그대로
   보여주면 어순이 어색할 수 있어 인용 표시 방식 설계 시 고려 필요.**

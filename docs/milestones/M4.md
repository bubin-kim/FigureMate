# M4 — Figure 설명 생성

> **비용 안내: 이 마일스톤은 M1.5의 Provider Layer를 통해 진행합니다.** 기본값
> (`LLM_PROVIDER=mock`)에서는 파이프라인 구조 검증까지 비용 없이 가능합니다. 다만 M4는
> **Vision 입력이 필수**이므로, "실제로 쓸만한 설명"을 확인하려면 M3와 마찬가지로 Ollama의
> Vision 모델(예: `llama3.2-vision`, `qwen2.5vl`) 또는 Anthropic으로 전환하는 품질 검증
> 단계가 필요합니다. 이 전환 여부와 시점은 M3와 동일하게 사람이 결정합니다.

## 목표 — 이 마일스톤이 끝나면 원래 목표한 5단계 파이프라인이 완성됩니다

```
업로드 → PDF 분석(M1) → Figure 추출(M2) → Figure↔본문 매칭(M3) → Figure별 설명 생성(M4)
```

M3가 만든 `FigureTextMatch`(근거 문단)를 입력으로, Vision 모델이 Figure 이미지를 보고
**근거를 갖춘 설명**을 생성합니다. `FigureExplanation`을 만들어 저장하는 것이 이 마일스톤의
전부입니다. M4는 뷰어 UI(M5)를 다루지 않습니다.

이 마일스톤은 M1~M3와 성격이 다릅니다. M1~M3는 "추출/매칭 로직을 정교하게 만드는" 결정론적
작업이 대부분이었지만, M4는 **처음으로 생성(generation)을 다루는 마일스톤**입니다. 생성은
검증 방식 자체가 다릅니다 — "정답과 일치하는가"가 아니라 "근거를 벗어나지 않았는가"와
"실제로 이해에 도움이 되는가"를 봐야 합니다.

## 이전 마일스톤에서 넘어온 조건 — 반드시 먼저 읽을 것

### M3 알려진 한계와의 연결 (가장 중요)

M3.md 하단에 기록된 대로, 다음 두 경로로 인해 **일부 Figure는 M4 시점에 매칭된 근거 문단이
전혀 없을 수 있습니다**:
1. 부록 Figure(paper1의 Fig 3~5 등)는 명시적 참조가 0건일 수 있음
2. Fig 3처럼 임베딩 유사도로도 정답 문단이 상위 후보에 들지 못하는 경우

**이 마일스톤의 가장 중요한 원칙**: 근거 문단이 없는 Figure에 대해, explainer_agent는
캡션과 이미지만으로 그럴듯한 설명을 지어내지 않습니다. `docs/data-model.md`의
`GroundingRef.source = "none_found"`를 실제로 사용하고, UI에 노출될 문구까지 프롬프트
단계에서 준비합니다 (예: "이 Figure에 대한 직접적인 본문 설명을 찾지 못했습니다. 아래
설명은 일반적인 지식에 근거합니다"). M3에서 "매칭이 없으면 없다고 말한다"는 원칙을 세웠던
것처럼, M4는 "근거가 없으면 없다고 말하면서 설명한다"로 이어받습니다.

### M2 알려진 한계와의 연결

캡션이 Figure 위에 있는 레이아웃이나 번호 없는 그림은 M2에서 이미 추출되지 못했으므로, M4는
이런 Figure를 아예 볼 수 없습니다. M3와 마찬가지로, M4의 채점 기준은 "M2가 추출한 Figure
대비"입니다.

### grounding_agent 스캐폴드 재사용

M3.md에서 grounding_agent가 "이후 모든 Agent의 원본"이 될 구조로 설계됐습니다. M4의
`explainer_agent`는 이 폴더 구조(`node.py`/`prompt.md`/`schema.py`)와 원칙(단일 책임,
근거 없는 생성 방지 장치, 골든 케이스 테스트)을 그대로 따릅니다. `docs/adding-an-agent.md`
체크리스트를 이번에도 동일하게 적용하세요.

## LangGraph 전환 — 왜 지금 하는가

`docs/architecture.md` 4절에서 LangGraph를 도입하는 이유로 든 것 중 "Figure 단위 병렬 처리"와
"부분 실패를 조건부 엣지로 표현"이 M3에서 이미 필요해졌었지만(순차 파이프라인으로 처리),
M4에서 Figure마다 독립적인 Vision 호출이 붙으면서 **병렬 처리의 실익이 커집니다** (Figure
6~7개를 순차로 Vision 호출하면 M3의 662초보다 더 길어질 수 있음 — 논문당 소요시간을 Step 1에서
먼저 추정하세요).

`services/pipeline.py`의 4단계(parse → figure → match → explain)를 LangGraph
`StateGraph`로 재조립하되:
- 앞 3단계(M1~M3)는 기존 로직을 그대로 노드로 감싸기만 합니다 (로직 재작성 아님)
- explain 단계만 Figure별로 `Send` API를 이용한 병렬 fan-out으로 구현
- `PipelineState`를 `docs/architecture.md`에 정의된 형태로 `app/graph/state.py`에 작성

이 전환이 기존 3단계의 결과(테스트 47개)를 깨뜨리면 안 됩니다 — 전환 후 회귀 테스트 전체
통과가 전제 조건입니다.

## 작업 지시 (Claude Code에게)

### Step 0. 설명 골든 케이스 준비 — M1~M3와 다른 방식

**이번엔 "정답 목록"이 아니라 "평가 기준(rubric)"을 만듭니다.** 설명 생성은 정답이 하나가
아니므로, M1~M3처럼 "이 문장이 정확히 일치하는가"로 채점할 수 없습니다.

1. `paper1.pdf`, `paper2.pdf`의 대표 Figure 3~4개(아키텍처형 1~2개, 플롯형 1~2개)를 골라,
   사람이 직접 "좋은 설명이라면 반드시 포함해야 할 요소" 목록을 작성합니다. 예:
   ```json
   {
     "figure_number": "Figure 1",
     "must_mention": ["encoder", "decoder", "self-attention", "residual connection"],
     "must_not_claim": ["논문에 없는 성능 수치를 지어내면 안 됨"],
     "grounding_expectation": "명시 참조 1건이 있으므로 grounding.source='paragraph'여야 함"
   }
   ```
2. 이 초안을 사람에게 보여주고 확인받은 뒤 Step 1로 진행합니다 (M1~M3와 동일한 검수 절차).
3. **이 rubric은 완벽한 자동 채점표가 아니라, LLM-as-judge 또는 사람의 최종 확인을 위한
   체크리스트입니다.** M1~M3의 "재현율 ≥90%"처럼 딱 떨어지는 숫자가 안 나올 수 있다는 것을
   미리 인지하세요.

### Step 1. 소요시간 사전 추정

Vision 호출 1건의 대략적인 소요시간을 Mock이 아닌 실제 환경(Ollama Vision 또는 Anthropic
중 사람이 지정하는 하나)으로 Figure 1개만 시험 호출해 추정하고, 논문당(Figure 5~7개 기준)
예상 총 소요시간을 계산해 보고하세요. 이 수치가 LangGraph 병렬화의 실익을 가늠하는 근거가
됩니다. **이 추정 없이 Step 6(LangGraph 전환)을 설계하지 마세요** — M3에서 실측 없이 설계를
진행하지 않는다는 원칙을 그대로 따릅니다.

### Step 2. 데이터 모델 확장

`docs/data-model.md`의 `FigureExplanation`, `ExplanationComponent`, `GroundingRef` 정의를
따라 구현하세요 (M1~M3와 동일한 패턴).

### Step 3. Figure 타입 판별

`app/services/figure_classifier.py`: Figure를 `architecture | plot | qualitative | other`로
분류. 휴리스틱(캡션의 키워드, 이미지의 종횡비 등) 우선 시도하고, 애매하면 Vision 호출 시
분류도 함께 요청하는 방식(별도 호출 없이 explainer_agent 프롬프트에 포함)을 고려하세요 —
Figure당 Vision 호출을 늘리지 않는 방향을 우선합니다.

### Step 4. `explainer_agent` 구현

`app/agents/explainer_agent/`에 M3의 `grounding_agent`와 동일한 3파일 구조로 구현하세요.

```python
def explain_figure(figure: Figure, matches: list[FigureTextMatch], figure_type: FigureType) -> FigureExplanation:
    """Figure 이미지와 매칭된 근거 문단을 입력받아 근거 있는 설명을 생성한다.

    matches가 비어있으면(M3에서 근거를 찾지 못한 경우), 근거 없음을 명시하는 설명을
    생성한다 — 절대 근거를 조작하지 않는다.
    """
```

**프롬프트 설계 원칙 (`prompt.md`)**:
- 단일 책임: "Figure를 설명한다"만. 발표 스크립트나 질문 생성 등은 범위 밖임을 명시.
- 입력에 매칭된 문단이 있으면 반드시 인용하고, 없으면 "본문에 명시되지 않음"을 명시하도록
  강제 (M3의 "확신 없으면 no" 원칙과 동일한 종류의 장치를 이번엔 생성 프롬프트에 적용).
- 아키텍처형 Figure는 `ExplanationComponent` 단위 분해를 시도, 그 외 타입은 `summary`와
  `detailed_explanation`만으로 충분.
- M3에서 발견한 "근거 인용 게이트"의 교훈(강한 게이트는 재현율을 죽이고, 없으면 오탐이
  생긴다)을 참고해, 이번에도 검증 강도를 실측 후 조정할 여지를 두세요.

### Step 5. 품질 검증 (사람 결정 지점)

M3의 Step 4~ 재측정 사이클과 동일한 방식입니다:
1. Mock으로 파이프라인이 끝까지 도는지 먼저 확인 (구조 검증)
2. 사람이 원하면 Ollama Vision 또는 Anthropic으로 전환해 실제 설명 품질을 확인
3. Step 0의 rubric 대비 결과를 사람에게 보여주고 판정을 받습니다. **M1~M3처럼 "오탐 여부
   판정 없이 넘어가지 않는다"는 원칙이 여기서는 "설명이 rubric을 충족하는지 판정 없이
   넘어가지 않는다"로 적용됩니다.**
4. 초기 시도가 부족하면(M3의 llama3.2 사례처럼) 모델 교체나 프롬프트 조정 옵션을 실측과
   함께 제시하고 사람이 결정합니다. 무비용 옵션을 먼저 소진한 뒤에만 유료 전환을 논의합니다
   (M3와 동일한 순서).

### Step 6. LangGraph 전환

위 "LangGraph 전환" 절 방침대로 구현. `app/graph/state.py`, `app/graph/pipeline_graph.py`

되면 정리 방향을 결정하세요(완전 삭제 vs 각 단계 함수는 노드 내부에서 재사용) — 이 판단
근거를 문서에 남기세요.

### Step 7. API + 파이프라인 통합

`app/api/routes_figures.py`에 `GET /papers/{id}/figures/{fid}` 응답에 설명 포함하도록 확장.

개별 Figure의 설명 생성 실패는 그 Figure만 "설명 실패" 상태로 표시하고 나머지는 계속 진행).

   paper2 Figure 1(매칭 2건)에서 3B Vision 모델이 제공된 근거 문단을 인용하지 않아 grounding이
   none_found로 폴백됐다. 이는 버그가 아니라 안전장치의 의도된 동작이다 — 모델이 근거를 안 쓰면
   코드가 근거를 지어내 채우지 않고(환각 차단), 대신 "본문 인용을 못 찾음"으로 정직하게 표시한다.
   → **M5 영향: UI에서 일부 Figure가 근거가 실제로 존재함에도 "본문에 근거 없음 — 일반 지식"
   배지로 보일 수 있다. 사용자에게는 "본문 인용을 못 찾음"으로 정직하게 보이므로 UX상 치명적이지
   않다.** 개선(무비용: qwen2.5vl:7b / 유료: Anthropic Vision)은 사람이 필요 시 전환.

2. **아키텍처형 설명은 응답이 길어 max_tokens가 크다(4096).** qwen2.5vl:3b가 verbose해
   1024·2048에서는 JSON이 잘려 파싱 실패했다(Step 5에서 발견·수정). components를 4~6개로
   제한하는 프롬프트 지시와 4096 상한으로 해결했으나, 더 복잡한 Figure에서 재발 가능성은 남는다.
   코드는 파싱 실패 시에도 none_found + 원문 일부로 죽지 않고 정직한 산출물을 낸다.
   → **M5 영향: 드물게 파싱 실패로 설명 품질이 낮은 Figure가 있을 수 있음. status로 구분 가능.**

## M4 설계 결정 사후 기록

("알려진 한계"가 아니라 의도적 설계 선택의 근거 기록. 채워지지 않은 기능적 공백은 위
"알려진 한계", 실측에 근거해 내린 결정은 여기.)

### explain은 Send fan-out이 아니라 단일 순차 노드로 구현 (Step 6)

M4.md Step 6은 explain을 Send fan-out으로 예시했으나, 실제로 구현·측정한 결과 순차 노드가
옳다고 판단했다. 근거:
- (a) sync LangGraph fan-out은 브랜치를 단일 워커 스레드에서 순차 실행해 병렬 이득이 없다(실측).
- (b) 여러 브랜치가 한 DB 세션에서 commit을 재진입 호출해 IllegalStateChangeError를 유발했다
  (테스트 5건 실패로 실제 발생 → 되돌림).
- (c) 로컬 단일 GPU ollama가 동시 호출에 marginal(0.71, Step 1 실측)해 애초에 병렬 이득이 없다.

따라서 순차 노드가 성능 동등·세션 안전·테스트 격리 유지 면에서 옳다. Figure별 로직은
`explain_stage_one`으로 분리돼 있어, async + 서버측 동시성 Provider(예: Anthropic)로 전환할 때
이 루프를 fan-out으로 바꿀 여지는 열려 있다(그때는 브랜치별 세션 필요). **성능·기능은 fan-out과
동일하므로 M5에 영향 없음.**

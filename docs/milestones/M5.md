# M5 — 최소 뷰어 UI

> **비용 안내: 이 마일스톤 자체는 프론트엔드 작업이라 API 비용과 무관합니다.** 다만 M5는
> 실제 논문 처리를 반복해서 트리거하며 화면을 확인하게 되므로, 로컬 `.env`의 `LLM_PROVIDER`가
> `ollama`로 설정된 상태라면 매번 업로드할 때마다 실제 Vision 호출(수 분 소요)이 발생합니다.
> UI 레이아웃/폴링 로직을 다듬는 반복 작업에서는 `LLM_PROVIDER=mock`으로 돌려두고, 최종
> 화면 확인 때만 `ollama`로 전환하는 것을 권장합니다 (전환은 `.env` 수정 후 백엔드 서버
> 재시작 필요 — M4에서 확인된 사항).

## 목표 — MVP 파이프라인 5단계의 마지막 조각

```
업로드 → PDF 분석(M1) → Figure 추출(M2) → 매칭(M3) → 설명 생성(M4) → 뷰어 UI(M5)
```

지금까지 만든 API(`/papers`, `/papers/{id}/figures`, `/papers/{id}/figures/{fid}`,
`/papers/{id}/figures/{fid}/image`, `/papers/{id}/figures/{fid}/matches`)를 사용해,
사용자가 **PDF를 업로드하고, Figure를 보고, AI 설명과 본문 근거를 함께 확인**할 수 있는
화면을 만듭니다.

M5는 "예쁜 디자인"이 목표가 아닙니다. **지금까지 만든 파이프라인의 결과물이 실제로
쓸모 있는 형태로 사람에게 전달되는가**가 목표입니다. 이번 마일스톤이 끝나면 README.md에
정의된 원래 MVP 범위가 전부 완결됩니다.

## 이전 마일스톤에서 넘어온 조건 — 반드시 먼저 읽을 것

### 본문 하이라이트는 M1의 `Paragraph.bbox` 품질에 직접 의존합니다

M1.md를 보면 `Paragraph.bbox`는 "있으면 사용"하는 선택적 필드로 설계되어 있었습니다. 지금까지
M1~M4 어느 마일스톤도 이 필드가 실제로 정확한 좌표를 담고 있는지 검증한 적이 없습니다
(M1의 테스트는 텍스트 재현율만 확인했습니다). **본문 하이라이트 기능을 만들기 전에 반드시
이 필드의 실측 품질을 Step 0에서 먼저 확인하세요.** bbox가 부정확하거나 비어있다면,
하이라이트는 "문단 전체를 배경색으로 표시"하는 수준으로 범위를 낮춰야 할 수 있습니다 —
이 판단은 실측 후 사람과 상의합니다.

### 처리 시간이 상당히 깁니다 — 폴링 설계에 직접 영향

M3(매칭)는 논문당 279~383초, M4(Vision 설명, qwen2.5vl:3b)는 Figure당 6~7초(전체
30~70초)이 실측되어 있습니다. 두 단계를 합치면 **논문 1편 처리에 5~7분** 걸릴 수 있습니다
(M4 실제 서버 검증에서 289초 실측). 폴링 UI는 이 정도 대기시간을 사용자가 견딜 수 있게
설계해야 합니다 — 단순 스피너로는 부족하고, 현재 어느 단계(`parsing` → `extracting_figures`
→ `matching` → `explaining` → `done`)인지 최소한 보여야 합니다.

### grounding "근거 없음" 표시는 M4에서 이미 준비됐습니다

M4의 `GroundingRef.note`에 사람이 읽을 수 있는 안내 문구("본문에서 이 Figure를 직접
설명하는 부분을 찾지 못했습니다...")가 이미 들어있습니다. M5는 이 문구를 그대로 표시하면
되고, 별도로 UX 문구를 새로 만들 필요가 없습니다.

## 범위

### 포함
- PDF 업로드 화면
- 업로드 후 처리 상태를 폴링으로 표시 (단계별 진행 상태)
- Figure 목록 화면 (썸네일 + 캡션)
- Figure 상세: 이미지 + AI 설명(summary/detailed/components) + grounding 표시
- 본문 패널: 매칭된 문단을 보여주고, bbox 품질에 따라 하이라이트 또는 문단 강조
- Figure 클릭 시 해당 매칭 문단으로 스크롤/포커스 이동

### 제외 (M6 이후, `docs/roadmap.md` 참고)
- 사용자 인증/계정
- 여러 논문 간 비교, 대화형 Q&A, 발표 스크립트 등
- 모바일 반응형 최적화 (데스크톱 우선, 반응형은 여유 있으면 시도)
- 세련된 디자인 시스템 — `frontend-design` 스킬을 참고하되 과도한 시간 투자 지양

## 작업 지시 (Claude Code에게)

### Step 0. 데이터 품질 실측 (+ 사람 확인)

UI를 짜기 전에, 실제로 보여줄 데이터가 화면에 담을 만한 품질인지 먼저 확인합니다.

1. 기존에 처리된 paper1(또는 재업로드)의 `Paragraph.bbox` 값을 실제로 몇 개 뽑아서,
   PDF 원본 페이지 위에 그 좌표가 실제로 해당 문단 위치를 가리키는지 육안으로 대조하세요
   (페이지를 이미지로 렌더링하고 bbox 사각형을 그려보는 간단한 스크립트로 확인).
2. 5~10개 문단을 샘플링해 정확도를 보고하세요. bbox가 대체로 정확하면 Step 5에서 실제
   하이라이트(사각형 오버레이)를 구현하고, 부정확하거나 없는 경우가 많으면 "문단 텍스트
   강조(배경색)"로 범위를 낮추는 것을 제안하세요.
3. 이 결과를 사람에게 먼저 보여주고, 하이라이트 방식(정밀 오버레이 vs 문단 강조)을 확인받은
   뒤 Step 1로 진행합니다.

### Step 1. Next.js 프로젝트 초기화

`frontend/`에 (README.md 안내대로):
```bash
npx create-next-app@latest . --typescript --app --tailwind --eslint
```

`docs/architecture.md`의 프론트엔드 스택(Next.js App Router, TypeScript)을 따릅니다.
`frontend-design` 스킬이 있다면 디자인 토큰/스타일 가이드를 확인하고 시작하세요.

### Step 2. API 클라이언트

`frontend/lib/api.ts`: 백엔드 엔드포인트를 감싸는 얇은 클라이언트 함수들.
```typescript
uploadPaper(file: File): Promise<Paper>
getPaper(id: string): Promise<Paper>
getPaperSections(id: string): Promise<Section[]>
getFigures(paperId: string): Promise<Figure[]>
getFigure(paperId: string, figureId: string): Promise<FigureDetail>
getFigureMatches(paperId: string, figureId: string): Promise<FigureTextMatch[]>
```
타입은 `docs/data-model.md`의 Pydantic 모델과 대응하는 TypeScript 인터페이스로 정의하세요
(백엔드 스키마가 바뀌면 이 타입도 갱신해야 함을 주석으로 남기세요 — 자동 생성 도구는
이번 범위에 포함하지 않습니다, 필요시 이후 마일스톤에서 고려).

### Step 3. 업로드 화면 + 폴링

`app/page.tsx` 또는 `app/upload/page.tsx`:
- 파일 선택 + 업로드 버튼
- 업로드 성공 시 `Paper.id`를 받아 상세 페이지로 이동
- 상세 페이지(`app/papers/[id]/page.tsx`)에서 `upload_status`가 `done` 또는 `failed`가
  아니면 일정 간격(예: 2~3초)으로 `GET /papers/{id}`를 폴링
- 현재 단계(`parsing`/`extracting_figures`/`matching`/`explaining`)를 사람이 읽을 수 있는
  한글 문구로 매핑해 표시 (예: "본문을 분석하고 있어요", "Figure를 찾고 있어요" 등)
- `failed` 상태면 `error_message`를 보여주고 재시도 안내

React 상태 관리는 기본 `useState`/`useEffect`로 충분합니다 — 이 규모에 별도 상태 관리
라이브러리는 과합니다.

### Step 4. Figure 목록 + 상세 화면

- `done` 상태가 되면 Figure 목록(썸네일 그리드 또는 리스트)을 표시
- Figure 클릭 시 상세 뷰: 원본 이미지, `explanation.summary`, `detailed_explanation`,
  `figure_type`이 `architecture`면 `components` 목록도 각각 표시
- `grounding`이 `none_found`면 M4가 준비한 `note` 문구를 눈에 띄지 않게(경고보다는 안내
  톤으로) 표시 — "이 설명은 일반적인 관찰에 근거합니다" 같은 뉘앙스

### Step 5. 본문 패널 + 하이라이트 연동

Step 0의 결정에 따라 두 가지 중 하나로 구현:

**(A) bbox 품질이 충분한 경우**: PDF 페이지를 이미지로 렌더링하거나 좌표 기반 오버레이로
매칭된 문단 영역에 하이라이트 박스를 그립니다.

**(B) bbox 품질이 부족한 경우**: `GET /papers/{id}/sections`로 본문 전체를 텍스트로
표시하고, 매칭된 `Paragraph`를 배경색으로 강조(하이라이트)합니다. 좌표 기반 오버레이보다
구현이 단순하고 안정적입니다.

어느 방식이든, Figure 상세 화면에서 매칭된 문단으로 스크롤 이동하는 상호작용을 구현하세요
(`GET /papers/{id}/figures/{fid}/matches`의 결과를 사용).

### Step 6. 통합 확인 + 문서 갱신

- 로컬 `.env`를 `ollama`로 전환하고 실제 논문 1편으로 전체 흐름(업로드 → 폴링 → Figure
  목록 → 상세 → 하이라이트)을 처음부터 끝까지 사람이 직접 확인
- `README.md` "프로젝트 상태"를 **MVP 파이프라인 5단계 전체 완료**로 갱신
- `docs/architecture.md` 프론트엔드 부분 실제 구현 반영
- 알려진 한계 기록 (M6 이후 기능과의 연결 — 예: 발표 스크립트 기능이 이 화면 위에 어떻게
  얹힐지)

## Step 0 결정 기록 및 알려진 한계

**하이라이트 방식 결정 (2026-07-13, 실측 후 사용자 확정): (B) 문단 강조.**
실측상 `Paragraph.bbox`는 품질이 높았으나(100% 채워짐, 육안 8/8 정밀, 완전 커버리지
paper1 98%·paper2 95%), "최소 뷰어" 취지와 프론트 구현 단순성을 우선해 (A) 정밀 오버레이
대신 (B) 문단 텍스트 배경색 강조를 채택한다. Step 5는 `GET /papers/{id}/sections`의 본문
텍스트를 표시하고 매칭된 `Paragraph`를 배경색으로 강조하는 방식으로 구현한다.

### 알려진 한계 (M5 완료 시점 최종 정리, 2026-07-14)

1. **일부 2단 컬럼 논문에서 문단이 원문과 다른 순서로 표시될 수 있다.** (B) 방식은 추출된
   문단 텍스트를 그대로 보여주는데, 컬럼 경계를 넘어 병합된 문단(paper1 ~2%, paper2 ~5%)은
   내부 어순이 뒤섞여 있다(예: "feedfor- Shortcut connections […] are those skipping…").
   **근본 원인은 M1 알려진 한계 #3(컬럼 경계에서 하이픈 분리 단어가 걸릴 때 병합 문단의 어순이
   뒤섞이는 파서 로직)이며**, M5는 그 텍스트를 그대로 노출하므로 이 한계를 승계한다. 화면에서
   해당 문단만 어색하게 보일 뿐 위치·매칭 자체는 정확하다. Step 5에서 실제 화면으로 확인함.
   근본 해결은 M1 파서의 컬럼 인식 개선이 전제이며 M5 범위 밖이다.

2. **"Figure를 찾고 있어요"(extracting_figures) 단계는 화면에 거의 표시되지 않는다.**
   Step 6 E2E 실측: 이 단계의 처리 시간(~1초)이 UI 폴링 주기(2.5초)보다 짧아, 폴링이
   parsing 다음에 바로 matching을 관찰한다. 버그가 아니라 타이밍 특성이며, 사용자 경험상
   문제 없음(오래 걸리는 단계는 전부 표시됨). 드라이버를 0.8초 폴링으로 돌려도 동일했다.

3. **Vision 설명 생성의 일시적(transient) 실패 가능성.** Step 6 E2E에서 마지막·최대 크기
   Figure의 Vision 호출이 타임아웃으로 1회 실패했다(모델 교체 시점 GPU 경합 추정).
   architecture.md 5절의 재시도 정책(지수 백오프 최대 3회)을 이 발견으로 실제 구현했고
   (`pipeline._retry_llm`), 재처리에서 "1차 timeout → 재시도 → 성공"으로 실증 복구됐다.
   3회 소진 시 해당 Figure만 status="failed"로 남고 UI에는 "설명이 없어요"로 표시된다
   (부분 실패 정책). 재시도는 현재 explain 단계에만 적용 — 매칭 단계 LLM 호출의 재시도는
   실패 사례가 관찰되면 같은 헬퍼로 확장한다.

### Step 6 E2E 실측 기록 (ollama 실데이터, 단일 브라우저 세션)

업로드 → 폴링(본문 분석 +2s → 매칭 +5s → AI 설명 +281s) → done(+513s) → Figure 목록
5개 → 상세(구성요소 6개, ollama 설명) → 본문 강조 2곳까지 한 세션에서 완주. 논문 1편
전체 처리 실측 ~8.5분(매칭 ~4.6분 + 설명 ~4분, 로컬 ollama 기준 — mock이면 수 초).
